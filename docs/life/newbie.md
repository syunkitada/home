# 新卒（およびメンター）のためのメモ書き

## 検索と質問と相談

- 課題や仕事を取り組んでいる中でわからないことがあったときにどうするか？
- 目星がついてる場合は、まずは自分で考え調べる
  - 目星がなければすぐ質問してよい
    - メンターは、新卒が自分で考えるべきものであれば、この辺から調べてみるとよいなどと道筋だけ示します
    - 先輩は忙しいからと躊躇する必要はない
      - 忙しければそう言います
      - 忙しいと言われても、ないがしろにされてるとは思わないように
  - やり方、使い方を調べる
    - ドキュメントがあるならまずはそこから
    - インターネットを検索する
  - 日本語より英語、時には英語より日本語
    - 英語で検索するほうが情報が引っかかりやすいので基本的には英語で検索するとよい
    - 英語に慣れてないなら日本語のほうが可読性が高いので、手軽に引っかかりそうなら日本語のほうがよい
  - エラーやログメッセージをそのまま検索して原因を調べる
    - それをほぼそのまま（環境情報などが含まれてるならそれを除外して）検索してみる
  - ソースコードを見る
    - ドキュメント整備が不十分なものに関しては、コードを読むのが一番早くて正確
  - 仮説を立て、実証する
    - 調べたものをそのまま信用するのではなく、実際にそれを検証する
    - 時間の制約がある場合や、実証が難しい場合はそのまま取り入れてもよいが、完全には信用しないこと
- 30 分以上は悩まなずに質問・相談する
  - 順調に課題をこなせてるのであれば質問や相談をする必要はない
  - メンターは新卒に付きっきりではないので、新卒からの質問や相談がなければ基本的に順調に課題を進めていると想定するので、課題に詰まったのなら相談する
    - とはいえ、言い出せない子もいるのでメンターもなかなか質問が来ないのであれば適度に聞く
    - 一日の終わりに振り返りをする
  - 質問するときは具体的に
    - 環境情報や、やったことなど
    - 自分なりに調べたこと考えたことも間違っていいので伝える
  - 正解は教えてくれない（正解をそのまま教えない、場合によっては教える
    - 仕事では正解を知ってる人が必ずいるとは限らない(まわりに相談しても答えが返ってこないことは多々ある)
    - 進捗との兼ね合いだが、なるべく新卒の力で正解までゴールする
  - 一言でいいので感謝を伝える
- 解決したらそれをまとめておく（順調に進んでてもちゃんとまとめる）
  - メンターは新卒がどのように課題を進めてたか、どう問題を解決したかを見たい
  - これがないと振り返りやアドバイスがやりずらくなる

## 質問は、詰問ではない

- 質問は質問であって詰問や叱責ではない
  - 裏の意図を勘ぐって「怒られている」と勘違いしないこと
- 質問を装って詰問してはいけない
  - とはいえオブラートに包む必要はなく、ダイレクトに聞いてよい
  - 言い方には気を付けること

## アウトプットとインプット

- 適度にアウトプットするのが大事、アウトプットしないと評価されない
  - 文章書いたり、プログラム書いたりなど
- うまくアウトプットできない時はインプットが足りない
  - アウトプットには 10 倍のインプットが必要
    - ドキュメント、類似コード、論文など
  - 調査や勉強や相談をもっとやろう
- とりあえずアウトプットしてみる
  - 設計も大事だがとりあえず作ってみるのも大事
    - 作り直しを恐れない
  - 車輪の再発明を恐れない
    - これを気にしてたらなにもできない
    - オリジナリティのある再発明はよいこと
      - 汎用的なものはネット上にあふれてるが、社内などの特定環境に特化したものは少ない
    - 勉強目的で再発明するのはよいこと
      - 仕様だけまねたり、既存コードもまねてみたりなど
    - 自分が思いつくことは、ほとんど誰かが先に思いついているは正しい
    - 既存でよくできてるものがあれば、それを使いまわすほうがよいのは正しい
    - 検索したら似たようなものがすでにあったというのはよくある
      - 作る前にとりあえず検索してみよう
- とりあえずインプットしてみる
  - 担当分野は積極的に勉強しておく
  - 担当分野でなくても、はてなブックマークなどで人気のやつをとりあえず読んでみる
    - 興味があまりなくても、知識を溜め込んでおくとどこかで役に立つ（たぶん）

## 自立して仕事を進める

- 部下は上司の下請けではない
- やりたい仕事、楽しい仕事は自分で作る
  - 上司から振られる場合もあるが、基本的には自分で手を上げないとできない
- やるべき仕事と、つまらない仕事は割り切ってがんばる
  - 上司から振られた仕事は、だいたいやるべき仕事
    - 中にはそうでないものもあるので、ちゃんとなんでその仕事をやるのか聞くこと
    - 納得できなければ反発することも大事
  - つまらないけど、やらないといけない仕事はある（割り切りも大事）

## モチベーションについて

- なにを取り組むにもモチベーションが大事
  - 特にテレワークでは、見られてるという感覚がないぶんサボる人はサボるので、よりモチベーションをもって取り組めるかというのが大事
  - 適度なコミュニケーションを行ったり、レポートを書いてもらって見られる意識を持ってもらう
  - 段階的なゴール設定でモチベーションを持続する
- 仕事を楽しむ
  - なんでも楽しんだほうが得
  - すべての仕事が楽しいわけではないが、 すべての仕事がつまらないわけでもない
  - 最初は楽しめなくても、できるようになったら楽しめる仕事もある

## 自信について

- 自信をつけるには、自分のスキルを磨いて高めるしかない
  - 誰かに教えてもらえる機会は少ないので、基本的には独力で頑張る
- 周りの人と比較して自信を無くさないように
  - 自分よりできる人がいる＝学ぶ機会がある、頼れる人がいる

## 時には逃げる事も大事

- 「やばいかな？違うかな？」と思ったら相談する
- 環境を変えることも大事（転職や配置換えなど)
  - ここにいても成長しないなと思ったら、環境を変えたほうが良い
  - 新しいことに挑戦するのは逃げではない
  - 退職する場合はあまり迷惑をかけないように
    - 多少迷惑がかかるのはしょうがないが、ちゃんとドキュメントや引継ぎ資料を残しましょう
- 生きるための逃げるのは良い
  - つらいことことを続けて鬱になるよりは逃げたほうがよい
  - 逃げたことは後悔しない、引きずらない、逃げて良かったと思うこと

## コピペ

- コピペすべきでないもの
  - ソースコード
    - 目当てソースコードを手に入れても中身を理解しないままコピペはしない
    - 理解できてればコピペしてもよいが、学習を目的とするなら写経すべき
  - コマンド
    - 目当てのコマンドを手に入れても挙動やオプションを理解しないまま利用しない
    - コマンドをいちいち検索するのは効率が悪いので、基本的にコピペはせずに手打ちしてコマンドを覚える
- コピペすべきもの
  - 間違いの許されないコマンド
    - 間違ったコマンドを実行すると事故に発展する可能性がある場合は、あらかじめ手順をまとめておきコマンドをコピペして実行する
  - ホスト名や IP アドレスといった環境情報
  - ログやエラーメッセージといったとくに身にならないもの

## 業務でよく使うツール類はなるべく速く使いこなせるようにする

- 今後の効率に大きく影響するので、業務でよく使うようなツール類にはプライベートの時間を使ってでも早めに慣れておく
- タイピング
  - タイピングの速さや正確さは仕事の速さや正確さに直結する
- vim
  - 端末操作でエディタを使うなら必須
  - 初学者は最低限 vimtuter だけはやる
- Visual Studio Code
  - コードリーディングで特に便利
- tmux
- zsh
- 英語

## メンターが用意しておくとよいもの

- 新卒が向けの環境設定用のコンテンツを用意する
  - チーム内ツールやメーリスなどのアカウントの登録とか
- 新卒が最低限学ぶべきコンテンツを用意する
  - チームで仕事をするうえで最低限知ってこないといけないこと
- 暇つぶし用の応用コンテンツを用意する
  - 課題を与えてそれが早めに終わっても暇にならないように応用コンテンツを用意しておく
